# ADR-0002: Drift + Supabase 双数据源策略

**日期**: 2026-06-20  
**状态**: accepted  
**决策者**: 项目团队

## 背景

项目需要同时支持:
- **离线优先**: 用户在没有网络时也能正常使用核心功能
- **云端同步**: 用户数据(收藏、评论)需要在多设备间同步
- **社区功能**: 需要实时的用户互动(发帖、评论、点赞)

单一数据源无法满足所有需求:
- 纯本地数据库无法实现云端同步
- 纯云端数据库离线时无法使用

## 决策

我们决定采用 **Drift (本地) + Supabase (云端)** 双数据源策略:

**数据分层**:
- **Tier 1 - 纯本地数据** (Drift): 
  - 水果数据、城市数据、推荐数据
  - 价格数据
  - 用户偏好设置
  - 这些数据不需要同步,从 GitHub Releases 定期更新

- **Tier 2 - 离线优先数据** (Drift + Supabase):
  - 收藏数据
  - 离线时存本地,有网络时双向同步

- **Tier 3 - 云端优先数据** (Supabase):
  - 社区帖子、评论
  - 用户认证信息
  - 这些数据需要实时性,离线时只读缓存

**同步策略**:
- 使用 Supabase Realtime 订阅变更
- 冲突解决: Last-Write-Wins (LWW)
- 本地数据作为 Single Source of Truth

## 备选方案

### 方案 A: 纯 Drift 本地数据库
- **优点**: 
  - 完全离线可用
  - 性能最优
  - 无需处理同步逻辑
- **缺点**: 
  - 无法实现社区功能
  - 无法多设备同步
  - 数据更新依赖应用更新
- **为何不选**: 产品需要社区功能和数据同步

### 方案 B: 纯 Supabase 云端数据库
- **优点**: 
  - 实时同步
  - 多设备一致性
  - 后台数据管理方便
- **缺点**: 
  - 完全依赖网络
  - 离线时无法使用
  - 性能受网络影响
- **为何不选**: 离线可用是核心需求,纯云端方案不满足

### 方案 C: 使用 Hive/Isar 替代 Drift
- **优点**: 
  - 性能更好
  - API 更简洁
- **缺点**: 
  - Drift 提供了更好的类型安全
  - Drift 的 SQL 查询更灵活
  - 项目已使用 Drift,迁移成本高
- **为何不选**: Drift 已满足需求,无需切换

## 后果

### 积极影响
- **离线可用**: 核心功能不依赖网络
- **数据同步**: 收藏等用户数据可以跨设备
- **社区功能**: 支持实时互动
- **灵活性**: 不同类型数据可以选择最适合的存储方式

### 消极影响
- **复杂度增加**: 需要维护两套数据源
- **同步逻辑**: 需要处理数据同步、冲突解决
- **成本**: Supabase 有使用成本(免费额度有限)

### 风险
- **数据不一致**: 本地和云端数据可能出现不一致
- **同步失败**: 网络问题可能导致同步失败

**缓解措施**:
- 引入 Repository 抽象层,隔离数据源变更
- 实现健壮的冲突解决策略
- 提供手动同步功能
- 监控 Supabase 使用量,避免超出免费额度
